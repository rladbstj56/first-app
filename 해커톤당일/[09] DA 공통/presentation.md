# 마케팅 성과 리포트 자동화 — 5분 발표 슬라이드 콘텐츠

> 용도: Claude 디자인(슬라이드 생성) 입력용. 슬라이드 1개 = 헤더 1개.
> 문장은 짧게, 표는 최소화, 숫자는 굵게 강조.

## 디자인 가이드 (톤·색상·스타일)

- 전체 톤: 깔끔하고 담백하게. 화려한 그래픽·과한 장식·불필요한 아이콘 지양.
- 색상: 무채색(흰색/회색/짙은 남색 또는 검정) 기본 + 포인트 컬러 1개만 사용(강조 숫자·핵심 문구에만). 색상 종류를 3개 이상 섞지 않는다.
- 레이아웃: 한 슬라이드에 메시지 1개. 여백을 충분히 두고 텍스트를 과밀하게 채우지 않는다.
- 표·숫자: 표는 꼭 필요한 최소 행만 유지, 핵심 숫자는 크고 굵게 강조.
- 폰트: 슬라이드당 폰트 크기 단계는 2~3단계(제목/본문/강조 숫자)로 제한, 서체 종류는 통일.

## Slide 2. 한 장 요약

- 문제: 매주 데이터 뽑고 → 계산하고 → 요약하는 수작업, 계산 오류 위험, "그래서 뭘 해야 하나"가 없는 리포트
- 해결: CSV 입력 → 정제 → 계산 → 이슈 탐지 → 진단 리포트 + 예산 재배분안까지 자동 생성, **같은 계산을 경영진용/마케팅팀용 2종 리포트로 독자에 맞게 렌더**
- 핵심 임팩트
  - 저효율 지출 **8,822,424원**(메타광고 과집행) 자동 탐지 → 사람이 놓치기 쉬운 "돈이 새는 지점"을 규칙 기반으로 짚어 회수 판단을 앞당김
  - 재배분 시 순증 매출 상한 **12,772,685원** 제안 → 탐지에서 끝나지 않고 "어디로 옮기면 얼마를 더 버는지"까지 구체적 액션으로 연결해, 추가 회의 없이 바로 결정 가능
  - 독자별 2종 리포트(경영진 요약본 · 마케팅팀 상세본) 자동 생성 → 같은 계산 결과를 승인권자(경영진)와 집행자(마케팅팀)가 각자 필요한 형태·깊이로 즉시 받음
  - 계산 전량 Python화 → 수식·셀 참조 오류 **0**

## Slide 3. 문제 정의 — 표면 문제 vs 진짜 문제

- 받은 문제: 5채널·8주 성과 데이터로 "어느 채널이 효율적인가, 무엇이 변했나"를 매주 보고
- 표면 문제: 리포트 작성에 시간이 오래 걸린다
- 재정의한 진짜 문제
  1. **매주 반복되는 수작업** — 매주 똑같은 계산을 사람이 손으로 반복 → 시간 소모 + 귀찮음 + 반복할수록 쌓이는 실수 위험
  2. **액션 없는 리포트** — 수치만 나열하면 받는 사람이 "그래서 이번 주에 뭘 해야 하는지" 알 수 없어, 리포트를 읽고도 다시 회의를 열어야 함
- 두 문제를 합치면 진짜 필요한 것은: **마케팅팀·경영진이 받은 자리에서 "이번 주 예산을 어디로 옮기고 무엇을 손댈지"를 결정하게 만드는 주간 의사결정 도구**
- 게다가 이 작업은 매주 반복되므로, 새 CSV만 갈아 끼우면 동일 품질로 재생성되는 **파이프라인**을 만드는 것이 매번 새로 만드는 것보다 압도적으로 효율적 → 이것이 이번에 "마케팅 성과 리포트 자동화 파이프라인"을 만든 이유
- 이 재정의가 이후 모든 설계 기준이 됨: **"받은 사람이 추가 회의 없이 이번 주 액션을 결정할 수 있는가"**

## Slide 4. 이 데이터가 실제로 만드는 함정 3가지

naive하게 "총매출·ROI 표"만 뽑으면 세 번 잘못된 결정으로 이어짐

1. **액션 부재** — 총매출 4.4억·전환 22,858건을 봐도 "그래서 뭘 하라는 건지" 없음 → 다시 회의를 열어야 함
2. **ROI 절대순위 오판** — 이메일 ROI 1위(1026%)라 증액이 당연해 보이지만, 시장 벤치마크 대비로는 개선여지 등급 → 잘못된 채널에 증액
3. **손실과 데이터 왜곡 혼동** — 이메일 매출이상치(2,884만원)를 손실로 읽으면 엉뚱한 예산 삭감으로 이어짐, 실제로는 원본 재확인이 필요한 데이터 오류

## Slide 5. 해결 접근 — 역할을 세 개로 쪼갬

- **계산 = Python** (정확성·재현성, LLM 수기 계산 배제)
- **해석 = 규칙 기반 + Claude** (판정 기준은 미리 코드로 고정, Claude는 완성된 숫자로 서술만)
- **반복 실행 = Skill** (매주 같은 품질로 재현)

| 고려한 선택지                 | 접은 이유                                 |
| ----------------------------- | ----------------------------------------- |
| Claude가 직접 숫자 계산       | 검증 불가·재현성 없음                    |

## Slide 6. 막혔던 지점 & 돌파 (문제해결력)

**① 급등/급락 판정 기준선 — "전주 대비"를 실제로 구현했더니 가짜 이슈가 나옴**

- **문제**: 이슈 판정 기준을 제공 문서(company-info.md) 문구 그대로 "전주 대비(WoW, ±50%)"로 구현
- **발견**: 카카오광고가 W4에 광고비를 실제로 줄였는데(딥), 다음 주 W5는 평소 수준으로 복귀했을 뿐인데도 "전주(W4) 대비 +52%"로 잡혀 **존재하지 않는 과집행**으로 오탐 — 딥 다음 반동(rebound) 구간이면 늘 발생하는 구조적 함정
- **돌파**: 기준선을 "전주 1개"가 아니라 "판정 주차를 제외한 나머지 7주의 중앙값(leave-one-out median)"으로 교체. 평균이 아니라 중앙값을 쓴 이유는 이상치 1건이 기준선 자체를 흔들지 않게(로버스트) 하기 위함 — 반동에도 흔들리지 않음을 실측으로 확인
- **판단 근거**: 제공 문서 문구를 그대로 따르는 게 아니라, 그 문구의 **의도**("진짜 비정상만 잡기")를 실측으로 검증하고 근거와 함께 벗어남 — 편차와 이유를 decisions.md에 기록해 "왜 문서와 다르게 했나"에 바로 답할 수 있게 함
- 의미: 스펙을 그대로 구현하는 것과 스펙의 **의도**를 지키는 것은 다를 수 있다 — 실측으로 반증하고, 벗어난 지점을 숨기지 않고 문서화하는 것이 재현 가능한 판단력

**② 재사용성 — 샘플 데이터에서만 잘 되던 코드**

- **문제**: 제공 샘플 데이터에서만 리포트가 잘 나왔음. 그런데 목표는 "매주 다른 데이터로도 동작"
- **발견**: 채널·주차를 줄인 가짜 데이터로 테스트하니, "손실 이슈가 하나도 없는 주"에서 리포트 생성이 멈춤(빈 결과 처리 실패) — 샘플 데이터만 봤으면 못 잡았을 함정
- **돌파**: 빈 경우(손실 0건·오가닉 없음 등)를 안전하게 넘기도록 수정 + 가짜 데이터 4종을 회귀 테스트로 고정. 이제 어떤 데이터를 넣어도 안 멈추는지 명령 한 줄로 검증
- 의미: "한 번 되는 코드"가 아니라 **"매주 안 깨지고 도는 파이프라인"**

## Slide 7. 전체 구조 (파이프라인)

```
성과 CSV
  → calculate.py     정제(중복 제거·결측 NaN 유지·이상치 동적 탐지) + 채널별 ROI·CTR·CVR·CPA 계산
  → detect_issues.py 규칙 기반 이슈 탐지(LOO 중앙값 기준선 + 적응형 임계값), 4레버 분류(예산/운영/데이터·트래킹/전략)
  → reallocate.py    run_pipeline — 재원(예산 레버 손실만) 산정 → 수혜처 선정 → 재배분안 생성
  → generate_report.py / generate_html.py   같은 결과를 독자별 2종(경영진용 exec_report / 마케팅팀용 insight_report)으로 렌더
  → verify_reproducibility.py   원본 + 변형 4종 자동 회귀 검증
```

- 핵심 원칙: **데이터 하나, 표현 여럿** — `run_pipeline()` 단일 계산 결과를 모든 문서(md·html)가 공유해 수치 불일치가 구조적으로 불가능

## Slide 9. 임팩트 (정직한 수치)

| 지표             | Before                       | After                            |
| ---------------- | ---------------------------- | -------------------------------- |
| 리포트 작성 시간 | 반나절 이상 (추정)           | 명령 한 줄                       |
| 계산 오류        | 수식·셀 참조 오류 상시 위험 | Python 고정 계산, 오류 0         |
| 산출물 깊이      | 수치 나열                    | 채널 3단 평가 + 이슈 7종 분류 + 우선순위 재배분안까지 완결된 의사결정 문서 |
| 새 데이터 대응   | 매번 처음부터                | 파일 교체만으로 동일 품질 재생성 |

- 저효율 지출 **8,822,424원** 자동 탐지, 재배분 시 순증 매출 상한 **12,772,685원**
- 매출 데이터 왜곡 2건(합 **30,223,909원**) 발굴 → 손실과 분리해 정직하게 "원본 재확인 대상"으로 표기
- 가정 고지: '반나절'은 추정, 순증 매출은 선형확장 가정의 **상한값**(실제는 하회 가능)

**산출물 깊이, 구체적으로 — "표 하나"가 아니라 "완결된 판단 체계"**

- 채널 평가만 3단으로 — 내부 ROI 순위(우리끼리 비교) · 시장 벤치마크 등급(업계 대비, 절대 1위의 함정까지 잡음) · 오가닉 성장 잠재력(미활용 자산)까지 한 채널을 세 각도로 본다
- 이슈는 7가지 유형(과집행·집행축소·성과급락·성과호재·매출이상치·매출결측·노출/클릭결측)을 규칙 기반으로 자동 분류하고, 다시 4개 조치 레버(예산·운영·데이터·전략)로 귀속해 "누가 손대야 하는지"까지 정해진 채로 나온다
- 손실 랭킹은 원(₩) 임팩트 × 발생 빈도로 우선순위 점수까지 계산돼 있어 "무엇부터"를 독자가 다시 판단할 필요가 없다
- 예산 재배분안은 숫자 하나가 아니라 근거(회수가능액·수혜처 벤치마크)·기대효과(순증 매출 상한)·실행 방식(소액 테스트→재측정→확대)·주의사항(선형확장 가정의 한계)까지 포함한 승인 가능한 완결 문서다
- 이 모든 계산 결과를 하나의 정본(run_pipeline)에서 독자별 2종(경영진 요약본·마케팅팀 상세본) × 2포맷(md·html), 총 4개 산출물로 동시에 생성 — 수치 불일치 없이 각자 필요한 형태로 즉시 받는다

## Slide 10. 잘한 점 정리

- **판단 기준을 문서화**: 결측·이상치·임계값·벤치마크 채택 등 모든 판단 근거를 decisions.md에 Step 단위로 기록 → "왜 이렇게 했나"에 즉시 답 가능
- **정직성 우선**: 손실과 데이터 오류를 구분, 수치를 부풀리지 않고 추정치는 추정이라 명시
- **재사용성 검증**: 하드코딩 제거 + 변형 데이터 4종 자동 회귀 테스트로 "실제로 재현되는지" 증명
- **독자 분리 설계**: 경영진(승인)과 마케팅팀(집행)의 필요를 같은 계산, 다른 렌더로 동시에 만족
