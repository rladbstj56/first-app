# [09] DA 공통 — 마케팅 성과 인사이트 리포트 자동화 (목표: Challenge 160점)

> **한 줄 요약: "숫자를 요약한 리포트"가 아니라, 마케팅팀·경영진이 받은 자리에서 '이번 주 예산을 어디로 옮기고 무엇을 손대지 말지'를 결정하게 만드는 주간 의사결정 도구를, 새 CSV만 갈아 끼우면 동일 품질로 재생성되는 파이프라인으로 만들었습니다.**
>
> - **정확성**: 합계·ROI·변화율·이슈 임팩트·예산 재배분을 전부 Python(`src/`)이 계산합니다. md·html 모든 표현물이 `reallocate.run_pipeline()` **단일 계산 결과**를 공유하므로 문서 간 수치 불일치가 0입니다. Claude는 완성된 숫자로 해석·서술만 합니다.
> - **실행력**: '핵심 이슈' 섹션의 근거를 **순위·근거·담당·시점**으로 취합한 '이번 주 업무 우선순위' 표 하나로 팀 업무 분배가 끝납니다. 예산 재배분안은 순증 매출 상한 약 **12,772,685원**까지 근거와 함께 제시해 별도 회의 없이 착수 가능합니다.
> - **독자 분리**: 같은 계산을 **경영진 요약본(exec_report, 승인용 1장)**과 **마케팅팀 상세본(insight_report, 실행용)** 두 고도로 렌더합니다. 계산은 하나라 수치가 어긋나지 않습니다.
> - **재현성**: 새 성과 CSV를 넣으면 정제 → 계산 → 해석 → 리포트가 자동 재생성됩니다. **매번 가장 최근 8주만** 분석하는 롤링 윈도우로 동향을 고정하며, 원본 + 변형 4종(12주 트림 포함)에서 두 리포트가 데이터에 맞게 생성되는지 **자동 검증 스크립트**로 증명합니다.
> - **신뢰성**: '예산으로 회수 가능한 손실'과 '데이터 정정 사안'을 분리하고, 채널 평가는 내부 순위가 아닌 **시장 벤치마크 대비** 정성 판정하며, 결측은 0으로 채우지 않고 집계에서 제외합니다. 모든 판단 근거는 decisions.md에 기록.

## 1. 문제를 어떻게 재정의했는가

**재정의.** 발제 원문은 "마케팅 성과 인사이트 리포트 생성"이지만, 실제로 풀어야 할 문제는 이것이었습니다.

> 숫자를 예쁘게 요약하는 게 아니라, **리포트를 받은 사람이 그 자리에서 '이번 주 예산을 어디로 옮기고, 무엇을 손대지 말아야 하는지'를 결정하게 만드는 것** — 리포트와 의사결정 사이의 간극을 없애는 것.

**왜 이렇게 봤는가 — 이 데이터가 스스로 증명하는 함정 3가지.**
naive하게 "총매출·ROI 표"만 뽑은 리포트는, 바로 이 데이터에서 세 번 잘못된 결정을 유도합니다.

1. **액션 부재** — 총매출 439,553,059원 · MER 862% · 전환 22,858건을 봐도 "그래서 이번 주에 뭘 하라는 건가"가 없습니다. 결국 리포트를 받고 **다시 회의를 열어** 액션을 정해야 함 → 리포트가 의사결정으로 이어지지 않음.
2. **ROI 절대순위 오판** — 이메일은 ROI 1025.98%로 **절대 1위**라 "증액하자"가 자연스러운 결론입니다. 그러나 시장 벤치마크(ROAS) 대비로는 **개선여지** 등급 → 절대 순위만 보면 잘못된 채널에 증액합니다.
3. **손실과 데이터 왜곡의 혼동** — 이메일 매출이상치 28,838,361원을 '손실'로 읽으면 엉뚱한 예산 삭감으로 이어지지만, 실제로는 원본 재확인이 필요한 **데이터 왜곡**(예산 손실 아님)입니다. 성격을 구분하지 못하면 조치가 정반대로 갑니다.

**Stakes — 안 풀면 무엇을 잃는가.**
이 세 함정을 방치하면 리포트는 "읽고 넘기는 문서"가 되고, 예산은 근거 없이 배분되며(잘못된 증액), 진짜 회수 가능한 기회손실(메타광고 과집행 8,822,424원)은 조치되지 않고 흘러갑니다.

**성공 기준 재정의.** 그래서 이 리포트의 '잘함'을 **"수치가 많다"가 아니라 "받은 사람이 추가 회의 없이 이번 주 액션을 결정할 수 있다"**로 바꿨습니다. 이 기준이 아래 모든 설계 결정을 지배합니다:
- 경영진(전체 성과·위험)과 마케터(채널별 실행 근거)를 한 리포트로 만족시키기 위해 **① 한눈에 보기 → ② 업무 우선순위 → ③ 이슈·채널 평가 → ④ 변화율** 순으로 구성.
- 함정 ②를 막기 위해 채널 평가를 **시장 벤치마크 대비**로 판정.
- 함정 ③을 막기 위해 이슈를 **'예산 회수 가능 손실' vs '데이터 정정 사안'으로 구조 분리**.

## 3. 핵심 산출물 — "이 표만 보면 이번 주 업무 분배가 끝난다"

리포트를 '숫자 요약'에서 그치지 않고, 받는 사람이 바로 손을 움직일 수 있는 **실행 문서**로 설계했습니다. 그 증거물이 [insight_report.md](output/insight_report.md)의 **이번 주 업무 우선순위** 표입니다 — '핵심 이슈' 섹션의 근거를 우선순위·담당·시점으로 한곳에 취합했습니다.

| 순위 | 할 일 | 근거 | 담당 | 시점 |
|------|-------|------|------|------|
| 1 | 메타광고 과집행 차단 후 회수분을 네이버광고에 재배분 | 초과지출 3,194,281원, 순증 상한 12,772,685원 | 마케팅 리드 | 즉시 |
| 2 | 이메일 매출이상치 원본 재확인·보정 | 왜곡·추정 28,838,361원 (예산 손실 아님) | 트래킹팀 | 이번 주 |
| 3 | 네이버광고 revenue결측 원본 재확인·보정 | 왜곡·추정 1,385,548원 (예산 손실 아님) | 트래킹팀 | 이번 주 |

(전체 6개 조치는 리포트 §이번 주 업무 우선순위 참조)

**예산 재배분 기획안**(마케팅팀 상세본 [insight_report.md](output/insight_report.md) §예산 재배분 기획안에 흡수)은 근거만 나열하는 데 그치지 않고, **문제 정의 → 재원(회수 가능 손실) → 재배분안 → 우선순위 판단 기준**까지 포함해 마케터가 추가 회의 없이 착수할 수 있게 설계했습니다. 우선순위 기준은 새로 만든 스코어가 아니라 채널 ROI·이슈 임팩트·시장 벤치마크 등급 등 **실제 계산값과 근거만** 사용했습니다. 경영진에게는 이 기획안의 **결정 요약(승인 요청)**만 [exec_report.md](output/exec_report.md)로 전달합니다.

**주차별 채널 효율 추이**(마케팅팀 상세본 §주차별 채널 효율 추이)는 8주 통합 순위가 못 보는 '주차 간 ROAS 역전·최근 상승 추세'를 드러내 예산 이동 **타이밍** 근거를 만듭니다. 정적 HTML 대시보드([insight_report.html](output/insight_report.html) · [exec_report.html](output/exec_report.html))는 md 리포트의 렌더링으로, 같은 수치를 보기 좋게 옮긴 보조 표현입니다.

## 4. 재현 가능한 파이프라인 구성

```
reallocate.run_pipeline  (정본 계산 1벌 — 롤링 8주 트림·집계·이슈·재원·재배분)
  -> calculate.py        (합계·ROI·CTR·CVR·변화율·주차별 ROAS 계산)
  -> detect_issues.py    (LOO 중앙값 기준선 + 적응형 임계값으로 이슈 자동 탐지)
generate_report.py  (같은 결과 → insight_report.md[마케팅팀] + exec_report.md[경영진])
generate_html.py    (같은 결과 → insight_report.html + exec_report.html)
verify_reproducibility.py (원본 + 변형 4종 자동 검증, 12주 롤링 트림 포함)
```

```bash
python3 src/generate_report.py [입력CSV] [출력MD]   # 마케팅팀 상세본 + 경영진 요약본 (md 2종)
python3 src/generate_html.py    [입력CSV]            # 위 둘의 HTML 대시보드 2종
python3 tests/verify_reproducibility.py             # 재현성 자동 검증(원본+변형 4종)
```

인자 생략 시 `data/marketing_performance.csv`를 읽어 `output/`에 생성합니다. Claude Code/Cowork에서는 `/marketing-report` 스킬 한 번으로 동일 절차가 실행됩니다. 새 CSV를 넣으면 최근 8주 창·변화율 주차·오가닉 섹션·재배분안 개수가 데이터에 맞게 자동 조정됩니다.

## 5. 주요 설계 결정 (방향을 바꾼 결정만 — 상세는 [decisions.md](decisions.md))

- **손실 이슈와 데이터 정정 사안 분리** — 초기에는 이상치·결측을 모두 '손실'로 묶었으나, 성격상 조치처가 다르다(예산 재배분 vs 트래킹팀 원본 재확인). `[예산]`·`[데이터·트래킹]`·`[운영]` 태그로 조치처를 명시하도록 전환.
- **채널 평가를 내부 순위에서 시장 벤치마크 대비로 전환** — 절대 ROI 1위(이메일)여도 자기 채널 시장 벤치마크로는 개선여지일 수 있어, 증액 판단을 industry-news 벤치마크 대비 정성 등급으로 함.
- **결측을 0으로 채우지 않고 NaN 유지** — 0으로 채우면 평균·ROI가 왜곡되므로, 집계에서 자동 제외하는 방식으로 수치 신뢰도 확보.
- **HTML을 이미지 차트가 아닌 정적 렌더링으로** — 과제 제외 범위(이미지 차트·웹앱)와 충돌하지 않게, Python 계산값을 CSS 요소로만 표시.
- **독자별 2종 리포트로 분리(계산 1벌, 렌더 2벌)** — 경영진(승인)과 마케팅팀(집행)의 필요 고도가 달라, 같은 `run_pipeline` 결과를 요약본·상세본으로 나눠 렌더. 정본이 하나라 수치 불일치가 구조적으로 불가능. 예산 재배분 기획안은 별도 파일 대신 마케팅팀 상세본에 흡수.
- **변화 비교는 두 축을 역할 분리** — 이상치·이슈 탐지는 강건한 '다른 주차 중앙값' 기준, 급변 모니터링은 '전주 대비(WoW)'. WoW만으로 이상치를 판정하지 않음(전주 노이즈 왜곡 방지).
- **최근 8주 롤링 윈도우 고정** — 8주 초과 CSV는 최근 8주만 분석해 동향을 고정(오래된 주차가 추이·기준선을 흐리지 않게).

## 7. 제약·금지 범위 (의도된 스코프)

과제 제약을 그대로 준수했습니다.
- **계산은 전부 Python** — 합계·ROI·변화율·재배분을 Claude 수기 계산에 위임하지 않음(감점 방지). Claude는 완성된 수치로 해석만.
