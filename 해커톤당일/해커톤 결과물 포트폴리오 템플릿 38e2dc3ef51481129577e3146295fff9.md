# 해커톤 결과물 포트폴리오 템플릿

<aside>
🎯

**독자 = 내가 푼 문제를 직접 낸 기업의 채용 담당자.**

이 사람은 ① 문제를 누구보다 잘 알고 ② 단순 심사가 아니라 *채용*을 검토하며 ③ 바쁘다. ④ 그리고 **기술 이해도는 낮다.**

모든 슬라이드/섹션은 "이 담당자가 지금 궁금한 게 뭔가"에 답하도록 구성한다.

</aside>

<aside>
⚠️

**이 자료의 범위 — 이력서·포트폴리오와 중복 금지.**

이력서·포폴은 **별도로 제출**된다(= "이 사람이 누구인가"는 거기서 본다).

이 자료는 오직 **"기업이 낸 문제를 어떻게 풀었나"** 하나만 다룬다.

경력·전반적 역량·자기소개는 **빼고**, 이력서/포폴이 답하지 못하는 *이번 문제 해결 과정*에 100% 집중한다.

</aside>

## 채용 담당자의 머릿속 질문 (이 순서가 곧 자료 순서)

1. 우리 문제를 **제대로 이해**했나? (대충 푼 건 바로 티 난다)
2. 실제로 **동작하는 걸** 만들었나? (데모 없으면 안 만든 것)
3. **어떻게** 거기까지 갔나? = 일하는 방식 (우리 환경에서도 이렇게 풀까?)
4. **효과**가 진짜 있나, 얼마나?
5. 우리 회사에 **적용 가능**한가? 한계는 정직한가?
6. **AI/도구**를 진짜 다루나? (AX 역량 = 핵심 차별점)
7. **같이 일하고 싶은** 사람인가? → *별도 자기소개로 답하지 않는다.* 문제 정의·막힌 지점 돌파·정직한 한계 등 **2~6번에서 일하는 방식이 그대로 드러나게** 한다.

## 템플릿 구조

### 0. 표지 — 5초 안에 "내 문제와 맞는 사람"인지

- 이름 + **한 줄 태그라인**: "〈기업 문제〉를 〈무엇으로〉 풀었다"
- (그 외 직군·역량·경력·연락처는 **이력서/포폴에 있으므로 생략**)

<aside>
💡

담당자는 이름보다 "내 문제를 푼 사람"으로 기억한다. 태그라인엔 기술명이 아니라 **기업이 낸 문제 + 해결된 결과**를 박아라.

❌ "n8n·GPT로 만든 자동화"  →  ✅ "지원자 이력서 검토 80분 → 8분으로 줄인 자동화"

</aside>

### 1. 한 장 요약 (Executive Summary) — 맨 앞 필수

> 가장 바쁜 담당자가 **이 한 장만 봐도** 판단되도록.
> 
- **받은 문제**: 기업이 제출한 P0 문제를 *원문 그대로* 인용
- **만든 것**: 한 문장으로 ("〇〇를 자동화/해결하는 △△")
- **핵심 임팩트**: 가장 강한 정량 수치 1~2개 (예: `처리시간 80분 → 8분`)
- **데모**: 30초 영상/GIF 썸네일 + 링크 (바로 누를 수 있게)

### 2. 문제 이해 — "우리 문제를 제대로 이해했나"

- **기업 제출 원문 문제** (그대로)
- **내가 재정의한 문제**: 진짜 페인포인트는 무엇인가 / 누가·언제·얼마나 자주 겪나
- **문제의 크기 (Before 수치)**: 지금 이걸 처리하는 데 드는 시간·비용·리스크
- **스코프**: 이번에 푼 것 / 의도적으로 안 푼 것 (정직하게 선 긋기)

<aside>
💡

담당자는 "문제를 떠먹여줘야 하는 사람"인지 "스스로 문제를 좁히고 정의하는 사람"인지를 여기서 본다. 원문을 *내 언어로 다시 쓴* 재정의가 가장 강한 신호.

</aside>

### 3. 해결 접근 & 의사결정 — "어떻게 일하는 사람인가"

> 결과보다 **재현 가능한 사고 과정**이 채용 신호다.
> 
- **접근 가설**: 왜 이 방식으로 풀기로 했나
- **버린 대안 + 버린 이유**: 고려했다 접은 선택지 (트레이드오프 사고 증명)
- **핵심 흐름**: 다이어그램 1장 — 단, *노드 이름이 아니라* **사람이 알아볼 동작**으로 ("메일 수신 → 자동 정리 → 점수 매겨 시트에 기록")
- **막혔던 지점 & 돌파 방법**: ★가장 중요★ — 실전 문제해결력은 여기서 드러남 (기술 난관도 "무엇이 문제였고 어떻게 풀었나"를 일상어로)
- **AI vs 나의 판단**: 어떤 부분을 AI에 맡기고, 어떤 결정은 직접 내렸나

<aside>
💡

"우리 환경에 데려다 놔도 이렇게 풀겠구나"를 상상하게 만드는 섹션. **읽는 사람은 기술 이해도가 낮다** — 흐름도는 비전공자도 고개 끄덕일 수준으로.

</aside>

### 4. 산출물 & 실증 — "진짜 동작하나"

- **데모 영상**: 핵심 기능이 실제로 돌아가는 장면 (실데이터로 한 사이클)
- **핵심 화면·결과물 스크린샷**
- **실제 동작 1건**: 입력 → 처리 → 출력 한 사이클을 실데이터로 ("이런 메일이 들어오면 → 이런 결과가 나온다")
- **무엇을 하는지만**: 담당자는 기술 이해도가 낮다. *어떻게 만들었나(노드·코드·모델명)는 빼고* **"무슨 일을 대신해 주는가"**로 설명. 깊은 기술 디테일은 전부 별첨으로.

### 5. 임팩트 — "효과가 있나, 얼마나"

- **Before / After 정량 표** (시간·비용·처리량·에러율)

| 핵심 지표 | Before | After |
| --- | --- | --- |
| (예) 처리 시간 | 80분/일 | 8분/일 |
| (예) 에러 | 휴먼에러 상시 | 0건 |
- **정성 효과**: 행동 변화, 리스크 제거, 누적 자산화 등
- **측정 방식 / 가정 명시**: 수치의 근거와 전제 (과장하지 않는 것이 오히려 신뢰)

### 6. 적용 가능성 & 아쉬움&비전 — "우리 회사에 들고 와서 쓸 수 있나"

- **우리 환경 적용 시 필요한 것**: 전제·연동·데이터·권한 (이 부분은 대답하기 어렵다면 생략)
- **개인적으로 문제를 해결하며 아쉬웠던 부분**: 정직하게 (못 한 걸 숨기면 더 크게 깎인다)
- **다음 스텝 / 확장 로드맵**: "2주 더 주어진다면" 무엇을 할까

<aside>
💡

정직한 한계 + 구체적 확장 비전 = 신뢰의 핵심. 완벽한 척하는 자료가 오히려 감점.

</aside>

### 7. 마무리 — "이 문제를 계속 맡기고 싶은가"

<aside>
⚠️

자기소개·경력·연락처는 **이력서/포폴에 있으니 반복하지 않는다.** 여기선 *이 문제*에 대한 마무리만.

</aside>

- **이 문제를 계속 맡았을 때의 그림** 한 줄 (예: "정식 합류 시 〇〇까지 확장 가능")
- **이 문제와 나의 연결고리** 한 줄 (왜 내가 이걸 잘 풀 수 있었는지 — 경력 나열 X, 이번 결과물 근거로)
- (그 외 일하는 방식·협업 스타일은 2~6번에서 이미 보여줬으므로 생략)

### (별첨) 기술&논리 상세

- 기술 : (ex) 워크플로우 구조 / 데이터 처리 과정 전체 구조 / 주요 노드 설명 / csv 파일 or 파이프라인 구조  / JSON·코드 / 추가 테스트 기록
- 논리의 영역은 비즈니스 언어(수치화, 구조화된 언어)로, 깊은 기술 검증을 원하는 담당자만 여기로.

## 포트폴리오(=발표) 흐름 매핑 (5분 발표 + 2분 Q&A)

| 시간 | 슬라이드 | 메시지 |
| --- | --- | --- |
| 0:00–0:30 | 0~1. 표지·요약 | "당신 문제를 이렇게 풀었습니다" + 데모 티저 |
| 0:30–1:30 | 2. 문제 이해 | 문제를 내 언어로 재정의 (담당자 신뢰 확보 구간) |
| 1:30–2:30 | 3. 접근·의사결정 | 어떻게 풀었나 + 막힌 지점 돌파 |
| 2:30–3:30 | 4. 산출물 데모 | **실제 동작 영상** (하이라이트) |
| 3:30–4:15 | 5. 임팩트 | Before/After 수치 |
| 4:15–5:00 | 6~7. 한계·확장·마무리 | 정직한 한계 + 적용 비전 + 이 문제와 나 |

<aside>
💡

**첫 1분 안에 데모를 보여줘라.** "되는 걸 만들었다"가 가장 빠른 신뢰. 설명이 길면 담당자는 "되긴 되나?"부터 의심한다.

</aside>

## 작성 원칙 (전 섹션 공통)

1. **읽는 사람은 기술을 모른다 — 비개발자 의사결정권자가 독자다.**
    - 도구·모델·노드 이름(n8n, GPT-4o, 〇〇 노드…)은 **본문에서 빼고** 별첨으로. 본문은 "**무슨 일을 대신해 주는가**"로만.
    - 전문용어는 한 번 풀어 쓰거나 **업무 비유**로 ("API 연동" → "두 프로그램이 서로 데이터를 자동으로 주고받게").
    - 판단 기준: *우리 부모님/영업팀장이 읽어도 이해되는가.* 이해 안 되면 임팩트도 0.
2. **모든 수치엔 근거·가정.** 측정 안 한 건 "추정"이라 밝힌다. 과장은 즉시 신뢰 손실.
3. **진행하며 아쉬웠던 부분 but 비전/확장하고 싶은 부분을 말해 진정성을 얻기** 안 되는 부분·미완성은 6번에서 정직하게 
4. **데모 = 산소** 영상/GIF 없으면 안 만든 것으로 읽힌다.
5. **분량보다 밀도** 슬라이드 한 장 = 메시지 하나.
